# AGENTS.md — wmaker-crm

This repository is the **C window-manager core** for the wmaker-ng
modernization (charter and full map live in `tacitness/wmaker-ng`:
[PLAN.md](https://github.com/tacitness/wmaker-ng/blob/main/PLAN.md) §2/§5/§8).
It is a **pristine, rebasable fork** of GNU Window Maker, tracked against
`repo.or.cz/wmaker-crm`. It is Layer 1: native, rebasable, upstreamable. All the
Rust/AI companions live in the separate `tacitness/wmaker-ng` repo — nothing AI
lives here.

```
upstream  https://repo.or.cz/wmaker-crm.git   (fetch only; push DISABLED)
origin    git@github.com:tacitness/wmaker-crm.git
```

## The one rule: keep master pristine

**master = upstream + a thin, mostly-new-files infra layer + (later) a small,
upstream-bound C patch series.** That is the whole invariant. Everything below
follows from it.

- **Never edit an upstream source file** except as a deliberate, documented
  commit in the patch series (see below). Infra (CI, docs, container, rebase
  tooling) is **new files only** — it must never touch the C tree, the autotools
  files, or anything that ships upstream.
- **Never push to upstream.** The `upstream` remote is fetch-only and its push
  URL is disabled. We contribute back via `git send-email`, not pushes.
- **Match upstream Window Maker C style exactly** for any source change — same
  indentation (tabs), brace style, naming, and comment conventions as the
  surrounding file. Run `./checkpatch.pl` on your diff.

## The patch series (actual code)

Real behavior changes are a **small series of focused commits** meant for
`git send-email` to the upstream list. Each commit:

- is the smallest seam that does the job — ideally a single hook point that
  dispatches into a *new* file, so a rebase rarely conflicts;
- is gated behind a config flag, disabled by default;
- carries a clear, upstream-style commit message (the goal is for upstream to
  *take* it);
- is **deleted from our series once upstream accepts it.**

The first such feature is tiling mode (see issue #5 / wmaker-ng PLAN §5/§8).

## The rebase ritual

We stay current with upstream by **rebasing our thin layer onto it**, never by
merging upstream into us (merges would dirty the "upstream + our commits"
shape).

```bash
make -f infra.mk rebase      # fetch upstream, rebase our layer onto upstream/master
```

`scripts/rebase-upstream.sh` does the work; `.github/workflows/upstream-sync.yml`
runs it monthly (and on demand), opening a PR on success or an issue with the
conflicting files on conflict. **It never force-pushes master.** While the fork
is pristine this is a clean fast-forward.

## Build & CI

The core is autotools. Top-level `Makefile` is **generated by `./configure`**
(and gitignored), so our infra targets live in `infra.mk`:

```bash
./autogen.sh && ./configure && make   # build Window Maker (the sacred build)
make -f infra.mk help                  # infra targets: rebase, image, run
```

`.github/workflows/validate.yml` proves the core still compiles on a clean
Ubuntu runner and runs a gitleaks secret scan. All third-party Actions are
pinned by commit SHA with a version comment — bump the SHA and the comment
together.

## Container base image

`make -f infra.mk image` builds a headless, scriptable Window Maker (Xvfb +
wmaker) from this source. It is the base layer the `ai-mcp` sandbox builds on —
the downstream consumers are tacitness/wmaker-ng#16 and #18.

## When you finish

State: summary, files touched, validation run, and confirmation that **no
upstream source file was modified** (infra in new files only).
